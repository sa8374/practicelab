# Отчёт о сверке "карточки ↔ контракт-пак"

## Обзор
Сверка концептов из каталога `concepts/` с контракт-паком (схема БД и OpenAPI).

## 1. Концепт: Работа ученика (student_work.md)

### Соответствие в схеме БД:
✅ **Таблица `student_works`** полностью отражает концепт:
- Все категории из карточки представлены как поля:
  - `execution_form` → "Форма выполнения" (устно, письменно)
  - `completion_time` → "Время выполнения" (in_lesson, homework, long_term)
  - `difficulty_level` → "Уровень сложности" (basic, advanced, creative)
  - `student_goal` → "Цель для ученика"
  - `engagement_level` → "Уровень вовлеченности" (formal, interested, motivated)
  - `independence_level` → "Степень самостоятельности" (independent, with_teacher, group, with_resources)
  - `processing_status` → "Статус обработки (администратор)" (uploaded, reviewing, graded, archived)
  - `content_type` → "Тип контента (администратор)" (text, image, audio, video, mixed)
  - `data_size` → "Объем данных (администратор)" (small, medium, large)
  - `access_level` → "Уровень доступа (администратор)" (student_only, student_teacher, public, group)
  - `storage_period` → "Срок хранения (администратор)" (temporary, long_term, permanent)
  - `learning_status` → "Статус обучения (администратор обучения)" (entry_task, current_work, final_work, additional_task)
  - `service_quality` → "Качество оказания услуг (администратор обучения)" (meets_standards, needs_improvement, exemplary)
  - `student_progress` → "Прогресс ученика (администратор обучения)" (beginner, progress, regress, stable)

✅ **Связи** отражают "Где живет" из карточки:
- `step_id` → связь с шагом (опционально)
- `lesson_id` → связь с уроком (опционально)
- `course_id` → связь с курсом (опционально)

### Соответствие в OpenAPI:
✅ **Endpoint `/student-works/{id}`**:
- Возвращает все поля из таблицы `student_works`
- Пример ответа включает все категории
- Статусы ответов: 200 (найдено), 404 (не найдено)

## 2. Концепт: Шаг (step.md)

### Соответствие в схеме БД:
✅ **Таблица `steps`** отражает концепт:
- `type` → "Минимальная единица взаимодействия" (read, answer, execute, submit)
- `has_checklist` → "Шаг с чеклистом самопроверки"
- `has_illustration` → "Шаг с иллюстрацией"
- `status` → категории статусов из карточки:
  - draft → "Шаг в черновике"
  - review → "Шаг на проверке"
  - published → "Опубликованный шаг"
  - hidden → "Скрытый шаг"
  - locked → "Заблокированный шаг"

✅ **Таблицы `checklists` и `checklist_items`**:
- Реализуют концепт "чеклист самопроверки"
- Связь `steps.has_checklist = TRUE` → существует чеклист

### Соответствие в OpenAPI:
✅ **Endpoint `/steps/{id}`**:
- Возвращает информацию о шаге
- Включает поля `has_checklist` и `has_illustration`
✅ **Endpoints для чеклистов**:
- `/checklists/{id}` - получение чеклиста
- `/checklist-items/{id}` - получение элемента чеклиста

## 3. Концепт: Урок (lesson.md)

### Соответствие в схеме БД:
✅ **Таблица `lessons`** отражает концепт:
- `has_review` → "Урок с ревью" / "Урок без ревью"
- `is_current` → "Текущий урок"
- `is_completed` → "Пройденный урок"
- `order_position` → определяет первый/последний урок
- `status` → категории статусов:
  - draft → "Урок в черновике"
  - review → "Урок на проверке"
  - published → "Опубликованный урок"
  - hidden → "Скрытый урок"
  - locked → "Заблокированный урок"

### Соответствие в OpenAPI:
✅ **Endpoint `/lessons/{id}`**:
- Возвращает все поля урока
- Включает `has_review`, `is_current`, `is_completed`

## 4. Концепт: Курс (course.md)

## Несоответствия и вопросы

### 1. Отсутствующие концепты:
- ❌ **Ревью (Review)** - упоминается в карточках, но нет отдельной таблицы
  - Реализовано через `student_works.processing_status = 'reviewing'`
  - Достаточно ли этого?
- ❌ **Прогресс ученика по курсу** - вычисляется через уроки
  - Нужна ли отдельная таблица прогресса?

### 2. Вопросы по реализации:
1. **Иллюстрации к шагам** (step.md):
   - Поле `has_illustration` есть, но нет таблицы для хранения медиа
   - Как хранить файлы иллюстраций?

2. **Версионирование контента** (step.md, lesson.md):
   - Упоминается "нельзя редактировать без создания новой версии"
   - Нет системы версионирования в схеме

3. **Связь "Работа ученика" с "Ревью"**:
   - Ревью реализовано как статус работы
   - Нужны ли отдельные таблицы для комментариев, оценок, обратной связи?

### 3. Технические детали:
- ✅ Все связи между сущностями отражены в схеме
- ✅ Все enum-значения соответствуют категориям из карточек
- ✅ OpenAPI endpoints соответствуют ресурсам в kebab-case
- ✅ Примеры данных включают все ключевые сценарии
